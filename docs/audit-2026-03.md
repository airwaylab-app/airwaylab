### Auth & CSRF Coverage

- 18/39 routes require Supabase session auth. All public routes are intentionally public (feedback, contact, contribution, stats).
- CSRF (origin validation) is applied on all POST routes. GET routes correctly skip CSRF.
- Stripe webhook uses signature verification (correct pattern).
- Auth tier validation has an explicit allowlist preventing escalation.
- No hardcoded secrets found anywhere in the codebase.
- No `.env` files committed to the repository.
- `SUPABASE_SERVICE_ROLE_KEY` is server-only (never in `NEXT_PUBLIC_*`).
- EDF parser runs in a Web Worker (browser only) â€” no server-side file parsing exposure.

### Overall Security Posture: **Strong**

The RLS coverage is comprehensive (every table has it enabled). The main gap is the `provider_interest` INSERT policy. Auth patterns are consistent. The rate limiting infrastructure is well-designed with Upstash Redis and in-memory fallback.

### Error Boundaries

`components/common/error-boundary.tsx` exists with proper Sentry integration. However, no test verifies the error boundary actually catches and renders fallback UI.

### Race Conditions

The `analysis-orchestrator.ts` has one known race: if two uploads trigger in quick succession, `clearIncrementalState()` in the second cancels partially-persisted nights from the first. This is by design, but no test covers it.
